Overview
v1.1 changes
What's new in v1.1 Three additions since v1.0:
· Partial refresh — refresh(bool partial) now accepts a flag. Pass true to perform a fast, flicker-reduced refresh that only redraws pixels that changed.
· App icons — Apps can now declare a launcher icon by overriding getIcon(), getIconW(), and getIconH().
· RLE image compression — The image() call now supports a compact 1-byte-per-run RLE format in addition to raw bit-packed data, reducing PROGMEM usage for typical icons by 50–80%.
What is NFX OS?

NFX OS is a minimal app runtime running on an ESP32-C3 with a 1.54" SSD1681 e-ink display. A host app manages the launcher and BLE connection. Developer apps plug in as App subclasses.

Sandboxing

Apps operate exclusively through NFXAPI*. Direct GPIO, CPU, memory, or storage access is intentionally withheld to keep apps safe and portable.

Display model

The display is 1-bit (black/white), 200×200 px. All drawing happens in an in-memory frame buffer. Call refresh() to push to hardware. Full refreshes take ~2 s; partial refreshes are significantly faster.

Color convention

c = 0 → white (paper). c = 1 → black (ink). All drawing calls default to black. Pass 0 to draw white (erase).

Coordinate system Origin (0, 0) is the top-left corner. X increases rightward, Y increases downward. The canvas is 200×200 pixels.

Writing Your First App
guide

An NFX app is a single .h file. You write that file — everything else is handled automatically by the platform. This guide walks through writing it from scratch.

All you need to deliver One .h file containing a class that inherits from App. The platform picks it up, compiles it, and shows it in the launcher automatically.
  • 1
    The file structure
    Your file must start with #pragma once and include "NFX_API.h". That include gives you access to the full NFXAPI interface and the App base class — those two things are everything your app will ever need.
    #pragma once
#include "NFX_API.h"

class AppMyThing : public App {
    // ... your app here
};
  • 2
    Implement the four required methods
    Your class must implement exactly these four methods. The OS calls them — you never call them yourself. Store the NFXAPI* pointer you receive in onStart as a private member — it is the only handle you need to draw, read input, send messages, and check time.
    class AppMyThing : public App {
private:
    NFXAPI* api;   // ← your only connection to everything

public:
    String getName() override {
        return "My Thing";  // shown in the launcher — keep it short
    }

    void onStart(NFXAPI* a) override {
        api = a;            // save pointer, draw initial screen
    }

    void onLoop() override {
        // called every ~10 ms — handle input, update, refresh
    }

    void onStop() override {
        // user exited — reset your state here
    }
};
  • 3
    Draw the initial screen in onStart
    onStart is called once when the user opens your app. Draw everything you need, then call refresh(false) — the false triggers a full e-ink refresh, which you always want on the very first draw.
    void onStart(NFXAPI* a) override {
    api = a;
    api->clear(0);                        // white background
    api->text(10, 10, "My Thing", 2);    // title at size 2
    api->line(0, 28, 199, 28);           // horizontal divider
    api->text(10, 40, "Press SEL", 1);  // instruction
    api->refresh(false);                  // full refresh — always on entry
}
  • 4
    Handle input and redraw in onLoop
    onLoop is called every ≈10 ms. The key pattern is a dirty flag: only redraw when your state actually changed. Use refresh(true) here — partial refresh is fast and only redraws what changed.
    int  count = 0;
bool dirty = false;

void onLoop() override {
    if (api->btnUp())   { count++; dirty = true; }
    if (api->btnDown()) { count--; dirty = true; }

    if (dirty) {
        api->clear(0);
        char buf[8]; itoa(count, buf, 10);
        api->text(80, 90, buf, 3);
        api->refresh(true);     // partial — only redraws what changed
        dirty = false;
    }
}
  • 5
    Reset state in onStop
    onStop is called when the user exits back to the launcher. Reset everything to its initial value so your app can be opened again cleanly without leftover state.
    void onStop() override {
    count = 0;
    dirty = false;
}
  • 6
    (Optional) Add a launcher icon
    Override getIcon() to show a custom icon in the launcher instead of a placeholder rectangle. See the App Icons section for how to encode the image data.
    static const uint8_t myIcon[] PROGMEM = {
    0x01, 0x7F, 0x7F, /* ... RLE data ... */
};

const uint8_t* getIcon()  override { return myIcon; }
int            getIconW() override { return 32; }
int            getIconH() override { return 32; }

App Skeleton

Copy this as your starting point. Every part is commented. Delete what you don't need.

App_MyApp.h — full template C++ 
#pragma once
#include "NFX_API.h"

class AppMyApp : public App {
private:
    NFXAPI* api;
    bool dirty = false;

    // ── Optional: RLE icon (see App Icons section) ──────────────
    static const uint8_t myIcon[] PROGMEM;

    // ── Private draw helper — keeps onLoop clean ────────────────
    void draw() {
        api->clear(0);
        // ... drawing calls ...
        api->refresh(true);   // partial for routine updates
        dirty = false;
    }

public:
    // ── Launcher label (keep short) ─────────────────────────────
    String getName() override { return "My App"; }

    // ── Called once on launch ───────────────────────────────────
    void onStart(NFXAPI* a) override {
        api = a; dirty = false;
        api->clear(0);
        // draw initial UI here
        api->refresh(false);   // full refresh on first draw
    }

    // ── Called every ~10 ms ─────────────────────────────────────
    void onLoop() override {
        if (api->btnUp())   { /* handle up   */ dirty = true; }
        if (api->btnDown()) { /* handle down */ dirty = true; }
        if (api->btnSel())  { /* handle sel  */ dirty = true; }
        if (dirty) draw();
    }

    // ── Called when user exits to launcher ──────────────────────
    void onStop() override {
        dirty = false;
        // reset any state here
    }

    // ── Optional icon overrides ─────────────────────────────────
    const uint8_t* getIcon()  override { return myIcon; }
    int            getIconW() override { return 32; }
    int            getIconH() override { return 32; }
};

// Icon data must be defined outside the class body
const uint8_t AppMyApp::myIcon[] PROGMEM = {
    0x01, /* ... RLE bytes ... */
};

App Lifecycle
getName()

Returns the label shown in the OS launcher below the icon. Keep it under ~12 characters to avoid clipping. The launcher uses SysFont_R16 for this label.

onStart(api)

Called once when the user launches your app. Store the api pointer, reset internal state, draw your initial frame, and call refresh(false).

onLoop()

Called every ≈10 ms by the OS main loop. Handle input, update state, and decide when to refresh. Never call blocking functions inside onLoop() — it blocks the entire OS.

onStop()

Called when the user holds UP+DOWN to return to the launcher. Release resources and reset state so the app can be relaunched cleanly without leftover values.

getIcon() — new

Returns a pointer to a PROGMEM byte array for the launcher icon image. Return nullptr (default) to show a placeholder rectangle. See the Images section for encoding.

getIconW() / getIconH()

Return the pixel dimensions of your icon data. The launcher renders icons at 64×64 — icons smaller than this are placed at their natural size. Default is 32×32.

Refresh cost E-ink full refresh takes ~2 s. Partial refresh is significantly faster but requires a prior full refresh to have occurred first. Use refresh(false) on first draw and refresh(true) for all subsequent updates.

Display Control
3 functions
void dispOn()
display
▶
Powers up the e-ink display hardware and initialises SPI. Called automatically by the OS at boot — apps generally don't need this unless they have explicitly called dispOff().
void dispOff()
display
▶
Sends the display into deep sleep (hibernate). The last rendered frame remains visible on the panel. Use for extended idle periods to save power. Wake with dispOn() followed by a full refresh(false).
void refresh(bool partial)
display updated v1.1
▶
Flushes the in-memory frame buffer to the physical display. The partial flag selects the refresh mode:
ParamTypeDescription
partialboolfalse → full refresh. Clears ghosting, moderate flicker, ~2 s. Use on first draw or after large content changes.
true → partial refresh. The OS automatically computes the bounding box of changed pixels and only redraws that region. Significantly faster and less flicker. Requires a prior full refresh.
Smart dirty-rect system When partial = true, the OS compares the new frame against the last rendered frame pixel-by-pixel to compute the minimal changed rectangle. If nothing changed, the refresh is skipped entirely — no flicker, no wasted time. The rectangle is byte-aligned (multiples of 8 px) to satisfy the SSD1681 hardware requirement.
Recommended pattern
void onStart(NFXAPI* a) override {
    api = a;
    drawFullUI();
    api->refresh(false);  // ALWAYS full on first draw
}

void onLoop() override {
    if (stateChanged) {
        updateCounterArea();
        api->refresh(true);   // partial for updates
    }
}

Image Format — Standard Bit-Pack

Format flag byte = 0x00. The simplest format: each pixel is one bit, packed MSB-first, row by row. Each row is padded to a full byte boundary.

Memory layout

For a W×H image: ceil(W / 8) × H payload bytes, plus 1 byte for the format flag. Bit 7 of each byte is the leftmost pixel of a group of 8. A 1 bit = black, 0 bit = white.

Structure
0x00format flag
→
0xF0pixels 0–7, row 0
0x3Cpixels 8–15, row 0
…rest of rows
Example — 8×2 checkerboard
// 0x00 = standard format
// Row 0: ■□■□■□■□  → 0b10101010 = 0xAA
// Row 1: □■□■□■□■  → 0b01010101 = 0x55
const uint8_t checker[] PROGMEM = {
    0x00,       // format: standard
    0xAA,       // row 0
    0x55        // row 1
};
api->image(10, 10, checker, 8, 2);
When to use standard format Use standard bit-pack when you have a tool that exports raw 1-bit bitmaps, or when the image has a very high density of alternating pixels (like a checkerboard) where RLE would expand rather than compress. For most solid or gradient-like icons, RLE is smaller.

Image Format — RLE Compression
new in v1.1

Format flag byte = 0x01. A compact run-length encoding format where each byte encodes both the color and the length of a run of pixels. Scanning order is left-to-right, top-to-bottom.

Byte structure — one byte per run

Each packet byte encodes the color in bit 7 and the run length minus 1 in bits 6–0. This means one byte can encode a run of 1 to 128 pixels of a single color.

BIT 7 COLOR
BIT 6 LEN[6]
BIT 5 LEN[5]
BIT 4 LEN[4]
BIT 3 LEN[3]
BIT 2 LEN[2]
BIT 1 LEN[1]
BIT 0 LEN[0]
color = (byte >> 7) & 0x01  |  run_length = (byte & 0x7F) + 1
Color bit (bit 7)

1 = black pixels in this run.
0 = white pixels in this run.

Length field (bits 6–0)

Stored as run_length − 1, so the value 0 means 1 pixel, value 127 means 128 pixels. Always add 1 when decoding.

Byte-by-byte walkthrough
0x01format = RLE
→
0x7F0 white × 128
0x841 black × 5
0x7F0 white × 128
0x831 black × 4
…continues
0x7F = 0b01111111 → bit 7 = 0 (white), bits 6:0 = 127 → run = 128 white pixels
0x84 = 0b10000100 → bit 7 = 1 (black), bits 6:0 = 4 → run = 5 black pixels
0x83 = 0b10000011 → bit 7 = 1 (black), bits 6:0 = 3 → run = 4 black pixels
How to encode manually (Python helper)
"""
Convert a 1-bit PIL image (or any pixel list) to NFX RLE bytes.
Pixels are 0=white, 1=black.  w, h must match the image dimensions.
"""
def encode_rle(pixels, w, h):
    result = [0x01]                   # format flag
    flat   = [p for row in pixels for p in row]
    i      = 0
    while i < len(flat):
        color  = flat[i]
        run    = 1
        while i + run < len(flat) and flat[i + run] == color and run < 128:
            run += 1
        packet = ((color & 0x01) << 7) | ((run - 1) & 0x7F)
        result.append(packet)
        i += run
    return result

# Output as C array:
# const uint8_t myImg[] PROGMEM = { 0x01, 0x7F, 0x84, ... };
Decoding tip — rows wrap transparently Runs are not constrained to a single row. A run of 200 white pixels can span two rows seamlessly. The decoder simply advances a pixel counter left-to-right, top-to-bottom and wraps at the image width. This is what allows very large uniform regions (like an all-white background) to compress into a single byte.
Size comparison A 32×32 icon (1024 pixels) in standard bit-pack always uses 128 bytes + 1 flag. In RLE it uses 1 byte per run — a typical icon with large uniform regions compresses to 30–60 bytes, a saving of 50–75%.

App Icons
new in v1.1

Apps can provide a launcher icon by overriding three methods in their App subclass. Icons are rendered by the launcher at 64×64 px — the selected app's icon is raised 20 px above the others.

const uint8_t* getIcon()  ·  int getIconW()  ·  int getIconH()
app new v1.1
▶
getIcon() — Returns a pointer to a PROGMEM byte array containing the icon image in either standard or RLE format. Return nullptr (the default) to show a placeholder shape in the launcher.

getIconW() / getIconH() — Return the pixel dimensions of the icon data. The launcher will draw the icon at its declared size; icons declared as 32×32 will render at 32×32 (centered in the 64×64 slot). Default is 32×32.
Icon placement in the launcher Icons are horizontally arranged with 74 px between centers. The currently selected icon is raised 20 px. Non-selected icons are shown at their natural vertical center (Y=80). Icons outside the screen edge are automatically skipped.
Complete icon declaration — inside a .h app file
class AppMyTool : public App {
    static const uint8_t myIcon[] PROGMEM;  // declared in class

public:
    const uint8_t* getIcon()  override { return myIcon; }
    int            getIconW() override { return 32; }
    int            getIconH() override { return 32; }
    // ... rest of app ...
};

// Definition must be outside the class body (PROGMEM + static member)
const uint8_t AppMyTool::myIcon[] PROGMEM = {
    0x01,  // RLE format
    0x7F, 0x7F, 0x47,   // 128+128+72 = 328 white pixels (top rows)
    0x85, 0x1A, 0x86,   // 6 black, 27 white, 7 black ...
    // ... more RLE data ...
};
PROGMEM + static member pattern Because of how Arduino's PROGMEM attribute works, the icon array must be defined outside the class body (not inline). Declare it as static const uint8_t myIcon[] PROGMEM; inside the class, then define it outside with const uint8_t AppMyClass::myIcon[] PROGMEM = { ... };. This pattern is shown in App_Clock.h as a reference. Alternatively, declare the array as a global in the .h file using an include guard (as in App_QuickSMS.h), but only if the file is never included more than once.

Three event types per button Each of the three physical buttons exposes three event functions: press (rising edge, fires exactly once), held (true every tick while held), and release (falling edge, fires exactly once). The OS calls sysApi.update() before your onLoop() every cycle, so state is always fresh.

#pragma once

    void draw() {
        api->clear();

        for (int i = 0; i < 5; i++) {
            int y = 24 + i * 20;
            if (i == sel) {
                api->rect(0, y-2, 199, y+12);
                api->text(8, y, items[i], 1, 0);
            } else {
                api->text(8, y, items[i], 1);
            }
        }
        api->refresh(true);   // partial — only selection bar moves
        dirty = false;
    }

public:
    String getName() override { return "Menu"; }

    void onStart(NFXAPI* a) override {
        api = a; sel = 0;
        api->clear(); draw();
        api->refresh(false);  // full refresh on entry
    }

    void onLoop() override {
        if (api->btnDown()) { sel = (sel + 1) % 5; dirty = true; }
        if (api->btnUp())   { sel = (sel + 4) % 5; dirty = true; }
        if (api->btnSel())  { /* open items[sel] */ }
        if (dirty) draw();
    }

    void onStop() override {}
};

#pragma once

class AppQuickSMS : public App {

    }

Guide — Animation Loop

E-ink is slow for full refreshes but partial refresh makes incremental animation practical. Key: only redraw what changed, and use refresh(true).

App_Spinner.h — loading indicator C++ 
#pragma once
#include "NFX_API.h"

class AppSpinner : public App {
    NFXAPI*  api;
    uint32_t lastFrame = 0;
    int angle = 0;
    static const int R        = 30;
    static const int CX       = 100, CY = 100;
    static const int FRAME_MS = 2500;  // one step every 2.5 s

    void draw(bool full) {
        float rad = angle * 3.14159f / 180.0f;
        int   ex  = CX + (int)(R * cos(rad));
        int   ey  = CY + (int)(R * sin(rad));
        api->clear();
        api->circle(CX, CY, R);
        api->circleFill(ex, ey, 5);
        api->text(66, 140, "Loading...", 1);
        api->refresh(full ? false : true);
    }

public:
    String getName() override { return "Spinner"; }

    void onStart(NFXAPI* a) override {
        api = a; angle = 0; lastFrame = a->now();
        draw(true);   // full refresh on entry
    }

    void onLoop() override {
        if (api->elapsed(lastFrame) >= FRAME_MS) {
            angle = (angle + 45) % 360;
            lastFrame = api->now();
            draw(false);  // partial refresh for steps
        }
    }

    void onStop() override {}
};
Tip — minimize changed area The partial refresh engine automatically computes the dirty rectangle. You can make it even smaller by only redrawing the changed region (e.g. erase just the old dot, draw the new dot) instead of calling clear() every frame. The smaller the changed area, the faster and smoother the partial refresh.
